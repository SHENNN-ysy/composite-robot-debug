# 节点编辑器纯前端完整集成实施方案

## 1. 实施结论

本阶段只做前端集成，保留 `node-editor-v2.html` 的完整功能，不建设后端。

建议采用：

> LiteGraph 编辑器内核复用 + React 页面壳集成 + 原有本地存储兼容 + 分阶段 TypeScript 化

不建议采用：

- `iframe` 嵌入：路由、主题、快捷键、登录布局和状态无法真正融合。
- 直接把整段 `<script>` 粘进 `FlowEditor.tsx`：会形成更难维护的超大组件。
- 一次性用 React Flow 重写：原型中的执行流、数据流、循环、子流程、代码生成和自定义节点很容易丢失。
- 一开始就大改数据格式：会导致已有项目 JSON 和 `localStorage` 数据无法恢复。

一期应继续使用 LiteGraph，并用“先无损迁移、再逐步重构”的方式接入。

## 2. 一期功能边界

必须保留以下功能：

### 2.1 编辑器基础能力

- 节点库分类、子分类、搜索。
- 点击或拖放创建节点。
- 节点连接、移动、选择、删除、复制。
- 执行流和数据流不同样式。
- 画布缩放、平移、适应视图。
- 自动布局和缩略图。
- 右键菜单。
- 节点备注和禁用。
- 撤销、重做。
- 快捷键。

### 2.2 节点与代码能力

- 流程控制节点。
- 通用机械臂节点。
- 法奥机械臂节点。
- PLC、视觉、AGV 节点。
- 变量、工具、数学节点。
- 节点参数来源：固定值、变量、端口。
- 节点方法名和 Lua 模板。
- 自定义节点新增、隐藏和恢复。
- Lua 代码生成、语法高亮、复制和下载。
- 图结构校验与问题定位。

### 2.3 项目能力

- 主流程和多个子流程。
- 子流程调用与引用。
- 变量池。
- 项目导入和导出。
- 浏览器本地保存和恢复。
- 兼容原型 `version: 3` 项目 JSON。

本阶段的“运行、暂停、停止、单步”仍只能是前端模拟或占位行为，不应描述成真实设备执行。

## 4. 具体迁移策略

### 4.1 第一步：建立功能基线

在修改代码前，保留原型作为对照基线：

```text
frontend/reference/node-editor-v2/
├─ node-editor-v2.html
└─ fixtures/
   ├─ basic-project-v3.json
   ├─ branches-and-loops-v3.json
   ├─ subflows-v3.json
   ├─ all-device-nodes-v3.json
   └─ custom-nodes-v3.json
```

至少准备以下回归项目：

1. 所有内置节点各一个。
2. 条件、循环、跳出和继续。
3. 主流程调用多个子流程。
4. 全局变量和流程变量。
5. 自定义节点、隐藏节点和自定义 Lua。
6. 导入后再次导出。

同时保存每个项目对应的 Lua 输出。迁移完成后逐字或按规范化结果比较，防止代码生成语义发生变化。

### 4.2 第二步：安装 LiteGraph

在 `frontend` 中安装与原型一致的版本：

```bash
npm install litegraph.js@0.7.18
```

第一阶段必须锁定 `0.7.18`，不要在集成过程中顺便升级 LiteGraph。等功能回归全部通过后，再单独评估升级。

在入口模块引入样式：

```ts
import 'litegraph.js/css/litegraph.css';
```

如果包的 CommonJS 导出在 Vite 中表现不稳定，创建唯一的适配文件：

```ts
// runtime/litegraphSetup.ts
import * as LiteGraphModule from 'litegraph.js';

export const LiteGraph = LiteGraphModule.LiteGraph;
export const LGraph = LiteGraphModule.LGraph;
export const LGraphCanvas = LiteGraphModule.LGraphCanvas;
```

项目其他模块只能从 `litegraphSetup.ts` 引用 LiteGraph，不能各自直接 import。这样可以确保所有节点注册和原型补丁作用在同一个单例上。

### 4.3 第三步：建立 React 画布生命周期

核心组件只负责创建容器和 canvas：

```tsx
export function NodeEditorWorkspace() {
  const containerRef = useRef<HTMLDivElement>(null);
  const canvasRef = useRef<HTMLCanvasElement>(null);
  const runtimeRef = useRef<EditorRuntime | null>(null);

  useEffect(() => {
    if (!containerRef.current || !canvasRef.current) return;

    const runtime = new EditorRuntime({
      container: containerRef.current,
      canvas: canvasRef.current,
    });

    runtime.mount();
    runtimeRef.current = runtime;

    return () => {
      runtime.dispose();
      runtimeRef.current = null;
    };
  }, []);

  return (
    <div ref={containerRef} className={styles.canvasArea}>
      <canvas ref={canvasRef} />
    </div>
  );
}
```

`EditorRuntime.dispose()` 必须清理：

- `graph.stop()`。
- LiteGraph Canvas 绑定。
- `window.resize`。
- `document.keydown` 和 `document.click`。
- 画布拖放事件。
- 缩略图 `setInterval`。
- 未完成的 `requestAnimationFrame`。
- 运行时对话框和悬浮菜单。

否则在 `/flow/program` 与其他路由之间来回切换时，会发生快捷键重复响应、定时器叠加和画布重复渲染。

### 4.4 第四步：先迁移运行时，不立刻重写算法

将原型的全局变量收进 `EditorRuntime`：

```ts
class EditorRuntime {
  private graph: LGraph;
  private graphCanvas: LGraphCanvas;
  private currentWorkflow = 'main_flow';
  private workflows = { main_flow: { data: null } };
  private variablePool = [];
  private customPaletteNodes = [];
  private hiddenSidebarTypes = [];
  private undoStack: string[] = [];
  private redoStack: string[] = [];
}
```

迁移原则：

- 先把原函数按原语义变为实例方法。
- 先消除 `window.appGraph` 和其他可变全局变量。
- 暂时不改变算法、不改项目格式、不改节点名称。
- 每迁移一组函数，立即用基线 JSON 验证。
- 完成功能等价后，再提取独立模块。

推荐迁移顺序：

1. LiteGraph 初始化与全局配置。
2. 节点工厂和所有内置节点。
3. 图序列化与恢复。
4. Lua 生成。
5. 校验。
6. 工作流和子流程。
7. 变量。
8. 自定义节点。
9. 导入导出。
10. 历史记录、布局、搜索、缩略图。
11. 属性面板、对话框和右键菜单。

### 4.5 第五步：节点注册必须幂等

原型会直接清空：

```js
LiteGraph.registered_node_types = {};
LiteGraph.searchbox_extras = [];
```

集成到整个应用后不应继续这样处理，因为它会清掉其他可能注册的节点。

改为：

```ts
let registered = false;

export function registerRobotEditorNodes() {
  if (registered) return;

  registerFlowNodes();
  registerRobotNodes();
  registerFrRobotNodes();
  registerPlcNodes();
  registerCameraNodes();
  registerAgvNodes();
  registerVariableNodes();
  registerUtilityNodes();
  registerMathNodes();

  registered = true;
}
```

React 开发模式可能触发重复 mount，节点注册、`renderLink` 补丁和连接校验补丁都必须保证只执行一次。

### 4.6 第六步：保留项目格式和本地数据

一期继续使用浏览器存储，但封装成统一接口：

```ts
interface ProjectStorage {
  load(): StoredProject | null;
  save(project: StoredProject): void;
  remove(): void;
}
```

建议统一新键名：

```text
robot-editor:project
robot-editor:autosave
robot-editor:preferences
```

同时兼容读取旧键：

```text
wf_index
wf_main_flow
wf_<流程名>
wf_variables
wf_custom_nodes
wf_hidden_sidebar
```

首次读取旧数据后：

1. 转换成完整 `version: 3` 项目对象。
2. 写入新键。
3. 暂时不删除旧键。
4. 显示“已迁移旧版本地项目”提示。

导入逻辑要继续同时支持：

- `version: 3` 多流程项目。
- 旧版单流程 LiteGraph JSON。

一期不应改变导出格式，避免用户无法回到原型工具。

### 4.7 第七步：React 化外围界面

画布内核稳定后，将以下区域改为 React 组件：

- 顶部工具栏。
- 节点侧边栏。
- 变量面板。
- 工作流标签。
- 属性面板。
- 代码预览。
- 校验结果。
- 状态栏。
- 模态框。

React 组件不直接修改 `graph._nodes`，只能调用 `EditorRuntime` 暴露的命令：

```ts
runtime.addNode(type);
runtime.saveProject();
runtime.generateCode();
runtime.validate();
runtime.undo();
runtime.redo();
runtime.switchWorkflow(name);
```

`EditorRuntime` 通过订阅接口通知 React：

```ts
runtime.subscribe((snapshot) => {
  setEditorSnapshot(snapshot);
});
```

快照只保存 UI 需要的信息，例如：

- 当前流程名称。
- 流程列表。
- 选中节点摘要。
- 节点和连线数量。
- 当前缩放比例。
- Lua 代码。
- 校验问题。

不要把完整 `LGraph` 或 `LGraphCanvas` 实例放入 React state 或 Zustand。

### 4.8 第八步：融入现有页面

现有路由 `/flow/program` 已经指向 `FlowEditor.tsx`，可以保持路由不变。

建议：

- 将现有 `FlowEditor.tsx` 替换成新的 `NodeEditorPage`。
- `ProcessOrchestration.tsx` 暂时不要继续复用同一个旧 `FlowEditorContent`。
- 如果“流程编排”暂时没有独立需求，将 `/flow/process` 重定向至 `/flow/program`。
- 页面继续使用现有登录保护和 `MainLayout`。
- 进入编辑器时允许折叠左侧主菜单，给画布留出空间。
- 不保留原型自带的第二套品牌 Header。
- 编辑器工具栏放在现有 `PageHeader` 下方。

### 4.9 第九步：隔离样式

原型中存在大量全局选择器：

```css
* {}
html, body {}
button {}
.app {}
```

这些样式不能直接放入现有 `global.css`。

处理方式：

- 编辑器自有样式全部放入 `NodeEditor.module.css`。
- 所有选择器以编辑器根容器为作用域。
- LiteGraph 官方 CSS 单独导入。
- 对 `.litecontextmenu` 等全局 LiteGraph 元素使用：

```css
:global(.litecontextmenu) {
  /* 编辑器主题覆盖 */
}
```

- 不修改 `html`、`body` 和全局 `button`。
- 原型 CSS 变量改为编辑器根容器变量，并复用现有主题色。

示例：

```css
.editorRoot {
  --editor-bg: #0f172a;
  --editor-panel: #1e293b;
  height: calc(100vh - var(--app-header-height));
  min-height: 600px;
  overflow: hidden;
}
```

### 4.10 第十步：解决 TypeScript 兼容问题

LiteGraph `0.7.18` 的类型声明可能不覆盖原型使用的内部字段，例如：

- `graph._nodes`
- `graph.links`
- `graphCanvas.ds`
- `node._disabled`
- `node._annotation`
- `node._paramSources`

不要在业务代码中到处写 `any`。创建集中扩展类型：

```ts
interface RobotGraphNode extends LGraphNode {
  _disabled?: boolean;
  _annotation?: string;
  _paramSources?: Record<string, ParameterSource>;
  _structure?: NodeStructure;
  _customMethodName?: string;
  _customScript?: string;
  getLuaCode?: () => string;
}
```

确实需要访问 LiteGraph 内部字段时，全部收口在 `runtime` 和 `graph` 目录，不向页面组件泄漏。

## 6. 本地保存方案

虽然暂时没有后端，也建议做以下增强：

### 6.1 自动保存

- 图发生变化后延迟 800～1500 ms 自动保存。
- 使用防抖，避免每次拖动都写 `localStorage`。
- 页面顶部显示“未保存 / 保存中 / 已保存”。
- 手动保存仍保留。

### 6.2 崩溃恢复

- 正式保存使用 `robot-editor:project`。
- 编辑过程使用 `robot-editor:autosave`。
- 启动时若自动保存时间更新，询问用户是否恢复。

### 6.3 存储容量

`localStorage` 通常容量有限。应捕获 `QuotaExceededError`：

- 保存失败时明确提示。
- 引导用户导出项目 JSON。
- 不允许静默丢数据。

### 6.4 数据校验

即使只有前端，导入 JSON 也需要：

- 限制文件大小。
- 校验顶层结构。
- 校验 workflows、variables、customNodes 类型。
- 对缺失字段应用默认值。
- 拒绝原型污染字段：`__proto__`、`prototype`、`constructor`。
- 捕获未知节点类型并展示兼容提示。

## 7. 功能回归验收

不能只以“页面能打开”为验收标准。

### 7.1 画布

- 路由首次进入正常。
- 离开再进入不出现重复事件。
- 浏览器缩放和窗口调整后画布尺寸正确。
- 创建、拖动、连接、删除、复制节点正常。
- 执行线和数据线样式正确。
- 右键菜单、快捷键和搜索正常。

### 7.2 节点

- 原型所有内置节点均可创建。
- 节点输入、输出、属性、默认值一致。
- 自定义节点可新增、导出、导入、隐藏和恢复。
- 节点结构动态修改后可以保存和恢复。
- 节点禁用和备注可以持久化。

### 7.3 流程

- 主流程不能删除。
- 子流程可新建、切换、删除和调用。
- 切换流程不会丢失未手动保存的画布状态。
- 子流程引用检查正确。
- 多流程项目导入导出无数据损失。

### 7.4 变量

- 变量新增、修改、删除、作用域和默认值正确。
- 变量节点选项实时刷新。
- 删除被引用变量时有提示。

### 7.5 Lua

- 基线项目生成结果与原型一致。
- 条件、循环、子流程、返回值缩进正确。
- 数据连接表达式解析正确。
- 自定义方法名与模板替换正确。
- 禁用节点不会生成执行代码。

### 7.6 本地数据

- 能读取原型旧 `localStorage`。
- 能导入原型 `version: 3` JSON。
- 能导入旧版单图 JSON。
- 刷新页面能够恢复。
- 存储失败不会静默覆盖或丢失。

### 7.7 构建

- `npm run build` 通过。
- TypeScript 严格模式下不新增散落的 `any`。
- 浏览器控制台无重复注册、空节点和事件泄漏错误。
- 编辑器代码通过路由懒加载，不增加登录页首屏负担。

## 9. 第一批代码改动建议

第一批提交只完成基础接入，不同时重构全部逻辑：

1. 在 `frontend` 安装并锁定 `litegraph.js@0.7.18`。
2. 新增 `features/node-editor/runtime/litegraphSetup.ts`。
3. 新增 `EditorRuntime` 和 `NodeEditorWorkspace`。
4. 把原型全部节点注册逻辑迁移到 `nodes` 目录。
5. 把原型项目导入导出和本地恢复迁入。
6. 用新页面替换 `/flow/program` 的旧 React Flow 页面。
7. 保留 `/flow/process` 原页面，暂不混改。
8. 建立 5 组基线 JSON 和 Lua 回归样例。
9. 确认路由往返、刷新恢复和生产构建通过。

完成这一批后，编辑器应已经能在现有项目中使用，并保留绝大多数原型功能。后续再把 DOM 拼接式侧栏、属性面板和对话框逐步改为 React，不影响图模型与项目数据。

## 10. 最重要的技术原则

1. 先保证功能等价，再重构代码风格。
2. LiteGraph 只初始化一次，所有全局补丁必须幂等。
3. 路由卸载时必须彻底清理事件和定时器。
4. 不同时维护两套机器人节点图模型。
5. 不改变原型节点类型字符串和 `version: 3` 项目格式。
